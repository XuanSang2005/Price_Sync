Price Sync — Kế hoạch Demo

Middleware đồng bộ giá HQ → POS (Oracle Retail Xstore). Mỗi đêm HQ POST một batch giá; hệ thống xác thực, kiểm định, ánh xạ, dựng file MNT rồi ghi vào thư mục Xcenter inbound.

Bản demo · môi trường: local (:5173 UI + :8080 API) hoặc Docker (docker compose up).

2. Kịch bản demo (chạy theo thứ tự)

# Thao tác Chứng minh tính năng Kỳ vọng thấy
1 Khởi động một lệnh. Chạy hệ thống (Docker up hoặc 2 terminal local). Mở Dashboard + Events. Đóng gói & triển khai · console vận hành Cả UI + API + Postgres lên. Dashboard/Events sạch (chưa có batch).
2 Happy path. POST 1 batch hợp lệ (Postman “202 - batch hợp lệ”). Đứng ở Dashboard xem. Intake (202) · pipeline async · ánh xạ MNT · console tự cập nhật (poll 1s) Tracker nhảy RECEIVED → PROCESSING → WRITTEN trong vài giây; Events + metrics tự đổi (không F5).
3 Xem sản phẩm ra lò. ls xcenter-inbound/ rồi mở file .mnt mới nhất. Dựng file DataLoader MNT (FHEAD/FDETL/FDELE/FTAIL) File có FHEAD,<date> · dòng FDETL,... (STORE→S, giá, tiền tệ, ngày) · FTAIL,<count>.
4 Bảo mật 4 lớp. Chạy các case Postman: sai API key · thiếu X-Timestamp · chữ ký rác · IP lạ. IP allowlist (403) · timestamp/replay (401) · HMAC-SHA256 (401) · API key (401) Mỗi case bị chặn đúng mã lỗi. Case “replay 10 phút” cũng 401 (chống phát lại).
5 Kiểm định & PARTIAL. POST batch trộn record hỏng (giá âm / currency sai). Mở trang chi tiết batch. 4 luật FR-03 · set-aside kèm lý do · PARTIAL / abort >20% Record hỏng = SET_ASIDE + lý do; phần còn lại vẫn ra file → batch PARTIAL. Nếu vượt ngưỡng (vd batch 1 record giá âm) → FAILED + email alert bay vào Mailpit (:8025).
6 Chống trùng. Gửi lại đúng batch vừa rồi (case “409 - batch trùng”). Rồi gửi bản version cao hơn. Idempotency 2 mức · supersede (bản mới thắng) Gửi trùng → 409 no-op; version cao hơn → bản cũ SUPERSEDED, chỉ bản mới ra file.
7 ★ ĐINH — Dynamic mapping. Tab Mappings → “Thêm luật”: promo_code, position 8, DIRECT. POST batch có "promo_code":"TET2027". Thêm field mới KHÔNG sửa code, KHÔNG redeploy File MNT mọc thêm cột ...,TET2027. Xoá luật → file về cũ. UI có ô “Xem trước” dòng output dựng từ luật.
8 Validate động. Khai field kiểu NUMBER (vd promo_pct). POST giá trị "abc". Kiểm hình dạng field khai thêm (STRING/NUMBER/DATE + required) Record đó SET_ASIDE lý do INVALID_PROMO_PCT; các record số hợp lệ vẫn qua.
9 Config-in-DB (không redeploy). Tab Config: đổi abort_threshold hoặc filename_pattern → Save. POST lại. Cấu hình sống trong DB, sửa qua console, hiệu lực tức thì Hành vi đổi ngay không restart: ngưỡng huỷ khác đi / tên file sinh ra theo pattern mới. UI cảnh báo nếu pattern thiếu .mnt.
10 Retry — tự chữa lỗi ghi. Chặn ghi: chmod a-w backend/xcenter-inbound → POST 1 batch. Sau đó mở lại: chmod u+w backend/xcenter-inbound. Ghi lỗi → PENDING_WRITE → retry backoff (6 lần, 30s→cap 10 phút) → WRITTEN/FAILED · operator re-drive Batch park PENDING_WRITE (retry_count + next_retry_at theo backoff); mở quyền lại → tick 15s tự retry → WRITTEN. Cạn 6 lần → FAILED → re-drive POST /api/v1/events/{id}/retry → 202 → chạy lại.
11 Reaper — cứu instance chết giữa chừng. Giả lập owner claim rồi chết 10 phút trước (psql):
UPDATE price_batch SET status='PROCESSING', owner_instance='dead-1', claimed_at=now()-interval '10 minutes' WHERE id=<id>;
Chờ ≤60s (reaper @Scheduled).		 Reaper thu hồi lease hết hạn (>5 phút) → active/active không kẹt batch khi 1 con chết Trong ≤60s reaper reset batch về RECEIVED (mất owner) → dispatcher claim lại với owner MỚI → chạy tiếp tới WRITTEN. Theo dõi: SELECT status, owner_instance, claimed_at FROM price_batch WHERE id=<id>; hoặc log app.
12 Nhật ký vòng đời. Trên trang chi tiết batch, xem mục “Processing history”. Audit log (bảng batch_log), timeline có mốc thời gian Chuỗi RECEIVED → PROCESSING → WRITTEN với timestamp từng mốc.
13 Sức khoẻ & mở rộng. Chỉ vào health strip ở Dashboard; nói về thiết kế 2 instance. Observability (health) · active/active an toàn (FOR UPDATE SKIP LOCKED + reaper) Đèn xanh API/DB + độ trễ. Giải thích: 2 instance chung 1 DB không xử lý trùng, một con chết con kia gánh.
Câu chốt khi demo: “Cùng một hệ thống chạy trong container — HQ bắn batch vào, đi qua 4 lớp bảo mật, được kiểm định & ánh xạ, rồi file MNT ra thư mục — tất cả tự động, không cần máy dev. Và muốn thêm một cột dữ liệu mới thì chỉ khai trên console, không đụng code, không deploy lại.”

4. Phụ lục

Cheat-sheet psql

psql price_sync                       -- nối DB local
\dt                                   -- liệt kê bảng
\d price_batch                        -- cấu trúc 1 bảng
SELECT id, batch_id, status, generated_at FROM price_batch ORDER BY id DESC LIMIT 10;
SELECT change_id, validation_status, set_aside_reason, extras
  FROM price_record WHERE batch_id = <id>;
SELECT record_type, position, json_field, rule_type, rule_value
  FROM mapping_rule ORDER BY record_type, position;
SELECT config_key, config_value FROM config;
\q

Gỡ rối nhanh

  • Startup chết “Could not resolve placeholder 'HQ_API_KEY'” → thiếu env var; chạy kèm HQ_API_KEY=… HQ_HMAC_SECRET=… ./mvnw spring-boot:run (fail-fast đúng thiết kế).
  • POST bị 403 → IP không trong allowlist. Docker thấy IP 192.168.65.1 → thêm vào ip_allowlist qua PUT /api/v1/config/ip_allowlist hoặc UI Config.
  • POST bị 401 → sai chữ ký/timestamp/API key → dùng Postman collection (tự ký).
  • UI không tự cập nhật → đang xem bản build cũ ở :8080; dùng :5173 (dev) hoặc rebuild.

Chưa có (nói thật nếu bị hỏi)

  • Đang chạy 1 instance; 2-instance active/active chưa dựng (cơ chế an toàn thì đã có).
  • Chưa có test tầng web tự động & Actuator/Micrometer; một vài nợ nhỏ backend (guard filename phía server, uq_change→409, secret ra .env cho prod).
